Known Concerns

Tech Debt

  • tai/commands/vastai.py is 425 lines with a 220-line up() function. The up command handles offer search, SSH key generation, instance creation, state persistence, SSH config writing, repo sync, bootstrap, and credential copy in a single function. It should be decomposed into a pipeline with named stages — each testable in isolation.
  • tai/core/sales/hnavi.py is 884 lines and contains 21 functions plus 7 classes covering login, job listing, negotiation fetching, message parsing, form submission, and DOM scraping. This is a God object. The scraping logic and domain models should be in separate modules.
  • tai/core/vastai/bootstrap.py embeds a multi-step bash script as an inline string template. The _INSTALL_SCRIPT_TEMPLATE constant is a 40-line bash program concatenated with Python string interpolation. Changes to the install flow require editing a raw string — no syntax checking, no tests that actually execute it.
  • The run_post_update() function in updater.py returns a 4-tuple of booleans (skills_ok, hooks_ok, templates_ok, style_ok) rather than a structured result. Any future extension requires adding another positional return value, breaking callers.

Fragile Areas

  • OAuth callback server uses a hardcoded port 8765. In tai/core/auth.py, _CALLBACK_PORT = 8765 is non-configurable. If the port is in use (another process, another tai login tab, a Docker mapping), the HTTPServer bind fails with an unhandled OSError — the user sees a Python traceback, not an actionable message. The server thread also has a hard 120 s timeout with no cancellation path if the browser never opens.
  • build_clone_cmd() in sync.py interpolates origin_url directly into a shell string passed to ssh ... bash -c. The URL comes from git remote get-url origin, which is user-controlled data. A URL containing shell metacharacters (spaces, semicolons, backticks) would break or be executed. The branch parameter has the same problem.
  • The keystore file fallback (secrets.json) is plaintext. The module docstring calls it "Encrypted file fallback" but _file_store writes unencrypted JSON to ~/.config/tai/secrets.json. Any process running as the same user, or a world-readable misconfiguration, exposes all stored credentials. The naming mismatch with the docstring makes this easy to miss.
  • wait_for_ssh() in provision.py contains a bare assert isinstance(data, dict). Python strips assert statements when run with -O. If the vastai API returns a list without an element, the subscript access data[0] is guarded but the assert is not — this is a control-flow dependency on optimisation flags.
  • hub.py has duplicated error-handling blocks. The file_upload and file_download commands manually re-implement the 401/4xx logic that _hub_request() already handles, because they bypass it. If the Hub auth model changes, these paths will diverge silently.

Missing Coverage

  • tai/commands/hub.py has zero tests. It was added in the most recent feature push and covers 10 commands (projects, tasks, pages, files, members, milestones, deliverables, search, summary, comment). All Hub API interactions, the project resolution logic, and file upload/download are untested.
  • tai/commands/agent.py and tai/core/agent.py have no test file. The run_parallel() and run_agent() functions, and the _build_codex_args / _build_gemini_args builders, have never been exercised by the test suite.
  • Sales modules (hnavi.py, aimitsu.py) are explicitly excluded from coverage via pyproject.toml [tool.coverage.run] omit. They contain the most complex browser automation logic in the codebase and are tested only via test_sales.py which mocks Playwright at a high level.
  • The OAuth callback server thread path is not tested. test_auth_core.py mocks at the HTTP level. The _run_callback_server function — including timeout behaviour and error capture — has no direct tests.

Security Surfaces

  • Shell injection via git remote URL in sync.py:build_clone_cmd(). origin_url and branch are read from git remote get-url origin and interpolated into a bash string sent over SSH. A compromised or malicious remote URL could inject arbitrary shell commands on the remote GPU instance. These values must be quoted with shlex.quote() before interpolation.
  • Asset name validation in updater.py is bypassable on Windows. _validate_asset_name() uses Path(name).name to strip directory components, but Path on POSIX treats \ as a literal character. A release asset named foo\..\..\evil.whl would pass validation on Linux/macOS but traverse on Windows.
  • User-supplied CLI arguments flow into subprocess commands without sanitisation. In agent.py, task.prompt is passed directly as a CLI argument to codex exec or gemini --prompt. While subprocess list form prevents shell injection, the agent itself will execute the prompt verbatim — there is no content-level guard.
  • ID token used as Hub Bearer token is never re-validated client-side after storage. hub.py retrieves the raw id_token JWT from keystore and forwards it as a Bearer header. Google ID tokens expire in 1 hour; the Hub client has no refresh logic — a stale token will silently produce 401 errors rather than an explicit re-login prompt.

Dependency Risks

  • inquirerpy>=0.3.4 — last PyPI release was 2022, no recent activity. The project appears unmaintained. It provides the interactive fuzzy-search prompts used in tai/core/prompt.py. Any Python 3.13+ compatibility issue or security patch would require a fork or replacement.
  • playwright>=1.40 is an optional dependency but is not installed in the default environment. The sales commands silently skip at import time, and the noqa: F401 guard in commands/sales.py masks the import failure. If Playwright's browser binaries are missing (installed via pip but playwright install not run), the error surfaces at runtime inside a try/except that prints nothing.
  • The package is not on PyPI. Bootstrap and update flows fall back to git+https://github.com/TrustedAI-CO/trusted-ai-cli.git. This means remote GPU instances pull directly from the default branch — there is no pinning. A broken commit to main will immediately break all new GPU provisions until reverted.
  • google-auth-oauthlib transitively pulls in requests while the rest of the HTTP stack uses httpx. Two HTTP clients are maintained simultaneously. The id_token.verify_oauth2_token() call uses google.auth.transport.requests.Request(), meaning token validation makes a synchronous requests call to Google's JWKS endpoint — invisible in the dependency graph and unable to share the httpx client configuration (timeouts, proxies).
