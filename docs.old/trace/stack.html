Technology Stack

Language

Python 3.11+
Chosen for its rich ecosystem of CLI and HTTP libraries, rapid iteration speed, and the team's existing fluency; the 3.11 floor specifically enables the standard-library tomllib (no extra dep for reading TOML config) and improved typing ergonomics.

CLI Framework

Typer
Drives the entire command surface (tai/main.py) because it turns type-annotated Python functions into fully documented, shell-completable subcommands with zero boilerplate, while the [all] extra pulls in Rich for coloured terminal output automatically.
Rich
Used directly for tables, progress spinners, and styled error messages throughout the commands; keeps output readable for humans without requiring the caller to format strings manually.
InquirerPy
Powers interactive prompts and fuzzy-search selectors (e.g. search_select in tai/core/prompt.py) for commands that need the user to pick from a list rather than type a free-form value.

Configuration & Secrets

TOML (stdlib tomllib + tomli-w)
Config is stored in ~/.config/tai/config.toml and optionally overridden by a per-project .tai.toml; TOML is preferred over JSON or INI because it is human-editable, comment-friendly, and strongly typed.
Pydantic + pydantic-settings
ProfileConfig and TaiConfig are Pydantic models so that config values are validated and coerced at load time, catching bad values before any command runs.
keyring
OAuth tokens and API secrets are stored in the OS-native credential store (macOS Keychain / Windows Credential Manager / GNOME Keyring) with a plain-file fallback for headless CI, meaning secrets never live in environment variables or on disk unprotected on developer machines.
python-dotenv
Allows .env files to inject TAI_* environment variables during development without polluting the shell permanently.

Networking

httpx
All outbound HTTP — to the TrustedAI internal API, Google OAuth endpoints, and external services — goes through a single build_client() factory (tai/core/http.py) that attaches a _BearerAuth flow capable of transparently refreshing the access token on a 401 and retrying once, so commands never have to manage token lifecycle themselves.

Authentication

Google OAuth 2.0 PKCE
Company identity is verified via Google Workspace; the PKCE flow (no client secret required) opens the system browser, spins up a transient local callback server on port 8765, and enforces the trusted-ai.co hosted domain at both the authorization URL (hd=) and JWT claim validation levels.

Document Generation

Typst
Branded PDF output (tai pdf) is compiled by the Typst binary rather than a Python PDF library; Typst's template language is more maintainable for design-heavy documents and produces smaller, higher-quality PDFs than HTML-to-PDF pipelines.

Browser Automation

Playwright (optional extra)
The sales command group uses Playwright for scraping Hnavi and Aimitsu sales portals; it is an optional dependency (pip install trusted-ai-cli[sales]) so the core CLI stays lightweight for users who don't need it.

External Services

TrustedAI Hub API (https://api.trusted-ai.internal)
The primary backend; manages projects, tasks, workspace members, and file assets — consumed by the tai hub command group.
Anthropic / Claude (tai ai, tai agent)
AI features are routed through the internal API (not the Anthropic SDK directly), keeping model selection and billing centralised; the default model is claude-sonnet-4-6 set in ProfileConfig.
vast.ai (tai vastai)
On-demand GPU provisioning for ML workloads is delegated to vast.ai via its CLI, wrapped by the vastai command group so engineers can spin up and tear down instances without leaving the terminal.
Google OAuth / Google APIs
Used solely for company identity verification; no Google Workspace data APIs are called by the CLI itself.

Build & Packaging

Hatchling + uv
Hatchling is the PEP 517 build backend; uv manages the virtual environment and lock file (uv.lock) for fast, reproducible installs in CI and on developer machines.
python-semantic-release
Automates version bumps and GitHub Release creation from conventional commit messages on every push to main, so release is a side-effect of a well-written commit rather than a manual step.

Testing

pytest + pytest-httpx / respx
Unit and integration tests run with pytest; pytest-httpx and respx mock httpx transports so network calls never escape the test suite, and the --cov-fail-under=80 flag enforces the 80 % coverage floor on every CI run.

Deployment Target

The CLI is distributed as a Python wheel attached to GitHub Releases and installed locally on developer machines via uv or pip. There is no server deployment — it is a developer-side tool only. A plugin entry-point mechanism (tai.plugins) allows internal teams to ship additional command groups as separate packages without forking the core.
