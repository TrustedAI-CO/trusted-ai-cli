Entrypoint & AppContext

What it does

The entrypoint is the skeleton that holds every command together. When you run tai pdf compile …, Typer parses global flags, runs the root callback to build a shared AppContext, then dispatches to the pdf subcommand. Every command receives the same context object and can read the active profile, verbosity setting, and loaded config from it — without knowing anything about how those were determined.

How it works

  1. Entry: The cli() function in tai/main.py is the [project.scripts] entry point. It enables debug logging if --verbose is present, then calls app(), catching any TaiError that bubbles up.
  2. Global callback: Typer runs the @app.callback decorated main() before every subcommand. It calls load_config() (see Config doc), builds an AppContext dataclass, and stores it in ctx.obj.
  3. Update banner: After building context, the callback checks a local cache for a newer version and prints a one-line nudge if one exists. This is fast — it reads a local JSON file, never hits the network on every invocation.
  4. Command dispatch: Each command group (pdf, hub, claude, etc.) is a separate typer.Typer() app registered with app.add_typer(). Single commands like login and setup are registered directly with app.command().
  5. Plugin discovery: After built-in commands are registered, _load_plugins() queries the tai.plugins entry-point group and adds any installed third-party Typer apps. This runs at import time, so plugins are available the same as built-in commands.
  6. Context access in commands: Any command that needs app-level data calls get_ctx(ctx) from tai.core.context. This is a safe wrapper — if for any reason ctx.obj is not an AppContext (e.g., in tests), it returns a default.

Key decisions

  • AppContext as a dataclass, not global state: Commands receive context through Typer's ctx.obj rather than module-level globals. This makes testing easy — tests can inject a pre-built AppContext and bypass the entire config-loading machinery.
  • Error handling at the boundary: TaiError and its subclasses carry a user-friendly message plus an optional hint. They are caught at the top level so every command just raises rather than duplicating print/exit logic.
  • Plugins via entry points, not a plugin directory: External extensions just declare a tai.plugins entry point; no config file needed.

Where to change things

  • tai/main.py — add new top-level commands, change global flags, or adjust plugin loading.
  • tai/core/context.py — add fields to AppContext that all commands should share.
  • tai/core/errors.py — add new error types or change exit codes.

Gotchas

  • The callback runs before every command, including --version. The version flag raises typer.Exit() early, before config is loaded, so it works even with a broken config.
  • If a plugin's entry point fails to load, tai prints a yellow warning and continues — it does not abort.
  • Tests that inject AppContext into ctx.obj skip load_config() entirely; make sure test contexts carry a valid config field if the command under test calls app_ctx.active_profile().
