Use HTML for Project Documentation

Context

TAI agents generate project documentation as part of the docs-init and docs-update workflows. Previously, agents wrote Markdown files which required a separate rendering step to view with proper formatting, navigation, and cross-document linking.

Markdown also lacks built-in validation — there is no way to enforce required sections, metadata, or structural rules without external tooling.

Decision

All project documentation under docs/ is authored as HTML. Agents write semantic HTML with data- attributes. A shared CSS stylesheet (_assets/style.css) handles all visual presentation. A shared JS runtime (_assets/docs.js) provides sidebar navigation, client-side validation, dark mode, and hot reload during development.

Agents do not concern themselves with styling. They write structure: 

<meta name="doc-type" content="decision">
<section data-section="context">...</section>
<section data-section="decision">...</section>

Validation rules are defined per document type (decision, design, spec, intent, etc.) and enforced both client-side via docs.js and server-side during docs-init / docs-update.

Consequences

  • Documents are browsable by double-clicking — no build step needed.
  • tai docs serve adds hot reload and sidebar navigation for iterative editing.
  • Validation is structural, not just linting — missing required sections are caught at authoring time.
  • Agents produce richer output: tables, code blocks, callouts, and cross-document links work natively.
  • Dark mode and print support come free from the shared stylesheet.
  • Trade-off: HTML is more verbose than Markdown. Mitigated by agents doing the writing, not humans.
Migration: Existing Markdown docs (README, ARCHITECTURE, etc.) remain at the repo root. Only docs/ uses the HTML system.
