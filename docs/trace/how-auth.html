Authentication

What it does

There are two independent auth flows. The company login (tai login) gets a Google OAuth token for the user's @trusted-ai.co account — this token is then used to talk to the Hub API. The Claude Code login (tai claude login) gets a refresh token from Claude.ai and passes it to the claude binary to authenticate the AI agent. Both flows use PKCE to avoid ever needing a client secret in the binary.

How it works — company login (Google)

  1. tai login calls tai.core.auth.login(). It generates a PKCE verifier (64 random bytes, base64url) and derives the challenge (SHA-256 of the verifier).
  2. A local HTTP server starts on localhost:8765 in a daemon thread, waiting for the redirect. The browser is opened to Google's auth URL with hd=trusted-ai.co (Google pre-filters non-company accounts) and the PKCE challenge.
  3. Google redirects to localhost:8765/callback?code=…. The server captures the code and stops.
  4. The code is exchanged for tokens at Google's token endpoint, sending the original verifier as proof. The returned id_token JWT is then verified with google-auth, and the hd claim is checked — both at the Google pre-filter level and in the JWT — to prevent domain bypass attacks.
  5. All five tokens (access, refresh, id, expiry, email) are written to the OS keychain via tai.core.keystore under the active profile name.

How it works — Claude Code login

  1. tai claude login generates its own PKCE pair and builds an auth URL for claude.ai/oauth/authorize. Because there is no local server to receive the redirect (the callback goes to platform.claude.com), the user is shown the URL, completes the browser flow, then pastes the callback URL or raw code back into the terminal.
  2. The code is exchanged at platform.claude.com/v1/oauth/token for a refresh token.
  3. The refresh token is injected as an environment variable (CLAUDE_CODE_OAUTH_REFRESH_TOKEN) and claude auth login is run as a subprocess. This hands off the final onboarding to the official Claude binary, so tai does not need to know the Claude internal token format.

Token refresh

When a command needs an access token, get_access_token() checks the stored expiry. If the token expires within 5 minutes, it refreshes using the stored refresh token — silently, in the background. If the refresh fails (network error or revoked token), it raises AuthExpiredError, which the error handler turns into a friendly "run tai login" message.

Key decisions

  • Dual domain enforcement: The hd parameter is set in the auth URL (Google pre-filter) and validated in the returned JWT. An attacker who intercepted the token exchange cannot supply a personal Google account token because the JWT would fail the hd claim check.
  • PKCE without a client secret: The OAuth client ID is embedded in the binary (this is intentional and safe for public desktop apps; the comment in the code says so). PKCE provides the security that a client secret normally would.
  • Claude login delegates to the claude binary: Rather than reimplementing Claude's session management, tai just provides the refresh token. This means Claude Code updates do not break tai auth.

Where to change things

  • tai/core/auth.py — all Google OAuth logic, PKCE helpers, token refresh.
  • tai/commands/claude.py — Claude Code OAuth flow (URL building, code exchange, subprocess handoff).
  • tai/commands/auth.py — the login, logout, whoami commands that call into tai.core.auth.

Gotchas

  • The local callback server uses a 120-second timeout. If the browser flow takes longer (slow network, user distracted), login will fail with "timed out" and the user must restart.
  • Port 8765 is hardcoded. If something else is using that port, login will fail with a bind error.
  • The id_token — not the access_token — is what the Hub API uses as a Bearer token. Make sure the keystore is storing and retrieving "id_token", not "access_token", when debugging Hub auth failures.
