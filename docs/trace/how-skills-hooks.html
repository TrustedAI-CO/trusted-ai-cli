Skills & Hooks

What it does

Skills are Markdown prompt files that Claude Code and Codex load as slash commands (e.g. /tai-review, /tai-ship). Hooks are Node.js scripts that Claude Code runs automatically at lifecycle events — before a Bash call, after a file edit, when a session ends. Both are bundled inside the tai package and installed into the right home-directory locations by a single command.

How skills work

  1. Source location: Bundled skills live in tai/data/skills/ inside the package. Each skill is a directory containing a SKILL.md file with YAML front matter (name, version, description) followed by the prompt body.
  2. Discovery: discover_skills(source_dir) walks the directory, finds every subdirectory that has a SKILL.md, and parses the front matter. The result is a list of SkillInfo dataclasses.
  3. Install — Claude Code: install_skills(source_dir) copies each skill directory to ~/.claude/skills/tai-{name}/. With --force, stale tai-* directories that no longer exist in the source are removed first, keeping the install directory clean.
  4. Install — Codex: The same function accepts a target=SkillTarget.CODEX argument, which points at ~/.codex/skills/ instead. The logic is identical.
  5. Dev mode fallback: When running from a git checkout (no package install), find_skill_source() walks up to the repo root and finds skills in .claude/skills/tai/. This means skills can be edited and tested without reinstalling the package.

How hooks work

  1. Definitions: tai/hooks/hooks.json is the manifest. Each entry has an event type (e.g. PostToolUse), a matcher (e.g. Edit|Write|MultiEdit), a description starting with [tai], and a command using ${SCRIPTS_DIR} as a placeholder.
  2. Resolution: resolve_hooks() replaces ${SCRIPTS_DIR} and ${LIB_DIR} with the absolute paths to the bundled scripts inside the installed package. This means hooks still work after the package is upgraded to a new path.
  3. Merge into settings: merge_hooks() reads ~/.claude/settings.json, removes any existing entries whose description starts with [tai], then appends the newly resolved hooks. Non-tai hooks are preserved. The result is written back atomically.
  4. Tagging: Every tai-managed hook has a description starting with "[tai]". This tag is the only marker used to identify and remove tai hooks — no separate manifest file.

Key decisions

  • Prefix naming (tai-*): All installed skills are prefixed so they never collide with user-created skills and are easy to identify and clean up.
  • Hooks are Node.js: Claude Code's hook runtime executes arbitrary shell commands. Node.js was chosen for the scripts because it is reliably available on developer machines (required anyway for many projects) and handles JSON parsing more cleanly than shell scripts.
  • Absolute paths at install time: Hooks resolve their script paths at install time, not at runtime. This is simpler and avoids PATH issues, but it means hooks must be re-installed after upgrading tai (the install path may change).

Where to change things

  • tai/data/skills/ — add or modify bundled skill prompt files.
  • tai/hooks/hooks.json + tai/hooks/scripts/ — add or change hook triggers and the Node.js scripts they run.
  • tai/core/skills.py — skill discovery and install logic; tai/hooks/__init__.py — hook resolution and settings merge logic.

Gotchas

  • Re-running tai claude setup-skills without --force skips already-installed skills. Pass --force to update existing ones after editing the source.
  • After running tai claude setup-hooks, Claude Code must be restarted for hooks to take effect — hooks are loaded at session start, not dynamically.
  • The [tai] tag in hook descriptions is load-bearing. Changing it will cause subsequent runs to duplicate hooks rather than replace them.
