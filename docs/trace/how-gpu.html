GPU Provisioning (vast.ai)

What it does

tai vastai up provisions the cheapest available GPU on vast.ai that meets a spec (GPU model, RAM, disk), generates a dedicated SSH keypair, waits until SSH is reachable, syncs the user's current repo to the instance, and records the instance under a local alias. tai vastai down tears it back down and cleans up the SSH config and state files. The whole flow is designed to be driven by the vastai-quick-gpu Claude skill without human copy-pasting.

How it works

  1. Offer search: tai.core.vastai.offers queries the vast.ai API (via the vastai CLI) for available GPU instances matching the requested spec, sorted by price. The first result becomes the target.
  2. SSH keypair generation: provision.generate_ssh_key(alias) creates an ed25519 keypair at ~/.ssh/vastai_{alias}_ed25519 if one does not already exist. The public key is uploaded to the instance at creation time.
  3. Instance creation: tai.core.vastai.provision.Provisioner calls vastai create instance … as a subprocess, passing the offer ID, SSH key, and disk size. It then polls SSH readiness — trying to connect every few seconds until the instance responds or a timeout is hit.
  4. SSH config entry: tai.core.vastai.ssh_config appends a Host block to ~/.ssh/config for the alias, setting HostName, Port, User root, and IdentityFile. This means the instance is reachable as just ssh {alias} from anywhere.
  5. State persistence: tai.core.vastai.state writes a JSON record (instance ID, alias, SSH details, creation time) to a file in ~/.tai/vastai/. This is how tai vastai list and tai vastai down know what exists without querying vast.ai every time.
  6. Repo sync: tai.core.vastai.sync uses rsync over SSH to copy the current working directory to the instance. It respects .gitignore patterns to avoid uploading build artefacts.
  7. Teardown: tai vastai down runs vastai destroy instance {id}, removes the SSH config block, deletes the local keypair, and removes the state record.

Key decisions

  • Provisioner as an injectable class: The Provisioner class accepts a runner (subprocess hook) and sleeper (time.sleep hook). Tests can substitute both to simulate instance creation and SSH polling without spending money or waiting.
  • Dedicated SSH key per alias: Using a separate keypair per instance makes teardown clean — deleting the alias deletes exactly its key, without touching the user's other SSH keys.
  • State file instead of always querying vast.ai: The vast.ai API has rate limits and latency. Storing instance metadata locally makes list and status instant, and means the CLI works for read operations even when the vast.ai API is slow.

Where to change things

  • tai/core/vastai/offers.py — offer query logic; change GPU filters or sort criteria here.
  • tai/core/vastai/provision.py — instance creation, SSH key generation, SSH readiness polling.
  • tai/core/vastai/sync.py — rsync options and gitignore integration.

Gotchas

  • The vastai CLI must be installed and authenticated separately (vastai set api-key …). tai does not manage vast.ai credentials — it just shells out to the binary.
  • SSH polling uses a fixed sleep interval. If a large disk image takes longer than the polling timeout to boot, the up command will fail even though the instance eventually starts. Increase the timeout or retry manually.
  • The state file is the source of truth for down. If it is deleted manually, tai vastai down {alias} will not know the instance ID and cannot destroy it — you would need to run vastai destroy instance {id} directly.
