Configuration & Profiles

What it does

The config system gives each engineer a single TOML file at ~/.config/tai/config.toml that controls API endpoints, AI model choice, company domain, and OAuth settings. Profiles let them switch between environments (dev, staging, prod) by name. A per-repo .tai.toml file can override any setting for that project. The entire thing is resolved and merged once at startup, before any command runs.

How it works

  1. Two TOML files are loaded: the user-level config at the XDG path ($XDG_CONFIG_HOME/tai/config.toml, defaulting to ~/.config/tai/) and an optional project-level .tai.toml in the current working directory.
  2. Shallow + deep merge: Top-level keys from the project file override the user file. Profile dictionaries are merged key-by-key, so a project .tai.toml can override just api_base_url inside the default profile without replacing all other keys.
  3. Pydantic validation: The merged dict is passed to TaiConfig.model_validate(). Unknown extra keys in a profile are silently allowed (extra = "allow"), which lets plugins add their own config keys to the same file.
  4. Profile selection: If --profile is passed (or the TAI_PROFILE env var is set), load_config() checks that the named profile exists and sets config.current_profile. If it doesn't exist, it raises ConfigError with a helpful hint.
  5. Active profile access: Commands call app_ctx.active_profile() which delegates to config.active() — returning the ProfileConfig for the selected profile, or a default ProfileConfig if nothing is configured.

Precedence (highest to lowest)

  1. CLI flags (e.g. --profile)
  2. Environment variables (TAI_PROFILE, TAI_JSON)
  3. Project config (.tai.toml in CWD)
  4. User config (~/.config/tai/config.toml)
  5. Hardcoded defaults in ProfileConfig

Key decisions

  • XDG compliance: Using $XDG_CONFIG_HOME means the config respects users who keep dotfiles organised (e.g. ~/.config/), rather than littering the home directory.
  • Pydantic for validation: Rather than hand-rolling type checks, ProfileConfig is a Pydantic BaseModel. Typos in TOML keys that don't match any field are silently allowed (via extra = "allow") so plugin keys don't cause startup failures.
  • Separate brand config: load_brand_colors() reads a brand.toml — a separate file used only by the PDF pipeline. This keeps brand identity (company name, hex colours) decoupled from API/auth config.

Where to change things

  • tai/core/config.py — add new config fields to ProfileConfig, change merge logic, or change default values.
  • tai/commands/config.py — the user-facing tai config get/set/list commands that read and write this file.
  • .tai.toml (repo root) — project-level overrides; can be committed to version control.

Gotchas

  • If ~/.config/tai/config.toml is missing, the CLI starts fine with all defaults — there is no hard requirement to run tai setup first.
  • Profile keys are case-sensitive. --profile Dev will not match a profile named dev.
  • save_config() writes the full config back, including any profiles that came from the project file. If you call it from a command, previously-project-only settings get permanently written to the user file — consider whether that is intentional.
