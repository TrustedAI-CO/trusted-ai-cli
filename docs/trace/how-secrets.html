Secret Management

What it does

The keystore module provides a single, consistent API for storing and retrieving sensitive values — OAuth tokens, API keys, anything that must not appear in config files or shell history. It tries the OS keychain first, falls back to a permission-restricted JSON file for headless environments like CI, and reads TAI_SECRET_* environment variables as a final read-only escape hatch.

How it works

  1. Retrieve — three layers: When any code calls keystore.retrieve(profile, name), it checks in order:
    1. Env var TAI_SECRET_<NAME> (uppercased, dashes replaced with underscores)
    2. OS keychain via the keyring library — macOS Keychain, Windows Credential Manager, or GNOME Keyring
    3. The JSON fallback file at ~/.config/tai/secrets.json
    If none of the three layers has the value, SecretNotFoundError is raised.
  2. Store: keystore.store() always tries the OS keychain first. If keyring raises KeyringError (e.g., no keychain daemon on a headless server), it transparently falls back to the JSON file. The JSON file is written with chmod 0o600 so only the owner can read it.
  3. Namespacing by profile: The keychain service name is tai-{profile}, so tokens for the default profile and the staging profile never collide.
  4. Rotate: keystore.rotate() simply overwrites the stored value. The keychain's own atomicity ensures the old value is not readable after the write completes.
  5. List names: keystore.list_names() returns secret names from the file fallback and any matching env vars. It intentionally never returns values.

Key decisions

  • Never in CLI flags: Following clig.dev guidelines, secrets are never accepted as command-line arguments. They are prompted for or read from the above sources only.
  • Env vars are read-only: You can inject a secret via TAI_SECRET_MY_KEY=…, but calling keystore.store() will not write back to the environment. This makes CI pipelines safe — they inject secrets via env and the binary never needs write access to the keychain.
  • JSON fallback is plaintext: The file is restricted to owner-read only (0o600), not encrypted. This is a deliberate trade-off: encryption would require storing a key somewhere, which just shifts the problem. The comment in the code explicitly documents this decision.

Where to change things

  • tai/core/keystore.py — the entire keystore implementation lives here; add new backends or change fallback order.
  • tai/commands/secret.py — user-facing tai secret set/get/rotate/exec commands.
  • tai/core/auth.py — calls keystore.store() and keystore.retrieve() directly for OAuth tokens.

Gotchas

  • On macOS, the first keyring write prompts for Keychain access. In automated tests, mock keyring.set_password or the test will hang waiting for a system dialog.
  • Env var names are uppercased and hyphens become underscores. A secret named my-api-key maps to TAI_SECRET_MY_API_KEY.
  • The JSON file fallback stores all profiles in one file. Deleting a profile from config does not automatically delete its secrets from the file — run tai secret commands to clean up manually.
