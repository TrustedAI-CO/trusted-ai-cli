PDF Generation

What it does

tai pdf compile turns a Markdown or Typst source file into a branded PDF. The pipeline loads a Typst template from a local install directory, optionally injects brand colours from a brand.toml, renders any Mermaid diagram blocks to SVG, then invokes the typst CLI to produce the final PDF. Templates and brand assets are bundled inside the tai package and installed once with tai pdf setup-templates.

How it works

  1. Template setup (one-time): tai pdf setup-templates discovers templates in tai/data/templates/ (each is a directory with a typst.toml manifest), then copies them to ~/.tai/templates/. Brand assets from tai/data/brand/ go to ~/.tai/brand/. With --force, existing templates are overwritten.
  2. Template resolution: When compiling, the user passes a template name. The code calls validate_template_name() (guards against path traversal), then looks up the template directory in ~/.tai/templates/{name}/. If it is not there, a TemplateNotFoundError is raised with a "run setup-templates" hint.
  3. Brand injection: If a brand.toml exists (in the project or user config dir), load_brand_colors() reads the company name and hex colour values. These are passed as Typst variables so the template can use #company-name and #primary-color without hard-coding them.
  4. Mermaid pre-processing: Fenced code blocks tagged as mermaid in the Markdown source are extracted, rendered to SVG by calling the Mermaid CLI (via Bun), and replaced with inline SVG references before the content reaches Typst. This happens in a temporary directory.
  5. Typst compilation: tai.core.typst checks that the typst binary exists and meets the minimum version requirement, then runs typst compile with the template as the root file and the processed content as input. Stderr from typst is surfaced as a TypstCompileError hint.

Key decisions

  • Typst over LaTeX: Typst compiles in milliseconds vs. LaTeX's multi-second runs, has a clean programmable syntax, and produces reproducible output. The trade-off is that users must install the typst CLI.
  • Templates installed to ~/.tai/, not the package: Templates can be edited by the user after installation (to customise for a client engagement, for example). Installing them to a user directory makes this natural without modifying the package itself.
  • Mermaid via Bun: The Mermaid CLI is a JavaScript tool. Rather than requiring a full Node.js + npm install, tai uses Bun's zero-install execution to run the CLI. This is why BunNotFoundError exists — Bun must be on the PATH for Mermaid to work.

Where to change things

  • tai/data/templates/ — add new Typst templates (each needs a typst.toml manifest).
  • tai/data/brand/ — default brand assets and the example brand.toml.
  • tai/core/typst.py — version checking and typst compile invocation; tai/core/mermaid.py — Mermaid SVG rendering.

Gotchas

  • The typst binary must be on the PATH. On macOS: brew install typst. The minimum required version is checked at compile time; too-old versions raise TypstVersionError.
  • Mermaid diagrams only work if Bun is installed (curl -fsSL https://bun.sh/install | bash). A document with no Mermaid blocks compiles fine without Bun.
  • Template names must match ^[a-zA-Z0-9][a-zA-Z0-9_-]*$. A name with slashes or dots will fail validation before touching the filesystem.
