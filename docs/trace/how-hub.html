Hub API

What it does

The tai hub commands let engineers manage workspace projects, pages, tasks, members, meetings, email (Gmail), calendar events, and project files through the internal Hub service — without opening a browser. Authentication reuses the Google id_token already stored in the keychain from tai login, with automatic refresh when near expiry, so there is no separate hub login step.

How it works

  1. Client construction: Every hub subcommand starts by calling _hub_client(ctx). This reads the active profile's hub_base_url (defaulting to https://hub.trusted-ai.internal), calls get_id_token() which auto-refreshes if the token is within 5 minutes of expiry, and returns an httpx.Client pre-configured with the Bearer token and a timeout.
  2. Request execution: Every API call goes through _hub_request(). It wraps client.request() and maps HTTP status codes to typed exits: 401 → "session expired, run tai login"; 403 → ExitCode.PERMISSION_DENIED; 404 → ExitCode.NOT_FOUND; 409 → ExitCode.CONFLICT. This keeps error handling consistent across all hub commands.
  3. Interactive selection: Some commands (e.g. picking a project when none is specified) call search_select() from tai.core.prompt. This is an is_interactive()- guarded fuzzy-search picker. In non-interactive mode (piped output, CI), the command falls back to requiring an explicit ID flag.
  4. Output formatting: All hub commands respect the global --json flag via app_ctx.json_output. In JSON mode, raw API response data is printed. In human mode, Rich tables are used for lists.

Key decisions

  • id_token as auth, not access_token: The Hub API validates the Google id_token (a signed JWT containing email and domain claims) rather than the OAuth access token. This lets the Hub server verify company domain membership without a separate user lookup.
  • --project accepts ID or UUID prefix: _resolve_project() accepts a full UUID or a unique prefix (e.g. first 6 chars). Ambiguous prefixes error with a list of matches. In interactive mode with no flag, a fuzzy-search picker appears.
  • Hub URL is a profile config key: The base URL is read from profile_cfg.hub_base_url so dev / staging / prod environments can point at different Hub instances by switching profiles.
  • Centralised error handling in _hub_request(): Rather than every command checking status codes, one function maps codes to exit codes. Adding a new error case is a one-line change in one place.

Where to change things

  • tai/commands/hub.py — all hub subcommands and the HTTP client helpers (_hub_client, _hub_request).
  • tai/core/auth.py (get_id_token()) — auto-refresh logic for id_token.
  • tai/core/config.py (ProfileConfig) — add hub_base_url as a typed field if you want default validation (currently read via getattr with a fallback).
  • tai/core/prompt.py — the interactive fuzzy-select helper used for project/task pickers.

Gotchas

  • The id_token auto-refreshes via get_id_token() when within 5 minutes of expiry. If the refresh token itself is invalid (e.g. revoked), Hub calls return 401 and the user must run tai login again.
  • hub_base_url is read via getattr(profile_cfg, "hub_base_url", None) because it is an extra key (not a declared field on ProfileConfig). This works due to Pydantic's extra = "allow" but means no type checking — consider declaring it explicitly.
  • The httpx.Client is used as a context manager inside each command. It is not shared across commands, which means each invocation opens a new connection. For bulk operations, refactoring to share a client would improve performance.
